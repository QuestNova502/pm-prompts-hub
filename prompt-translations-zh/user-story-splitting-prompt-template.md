## 用户故事拆分案例逻辑：
1. 它是否包含多个工作流步骤？如果是，则沿工作流步骤拆分故事。
2. 它是否引入业务规则变化？如果是，则沿业务规则变化拆分故事。
3. 它是否展示数据变化？如果是，则沿数据变化拆分故事。
4. 它是否具有复杂的验收标准？如果是，则沿离散的"when"行动或"then"结果拆分故事。
5. 它是否需要重大努力来构建？如果是，则沿重大努力里程碑拆分故事。
6. 它是否包含外部依赖关系？如果是，则沿外部依赖关系拆分故事。
7. 它是否需要重大的 DevOps 努力？如果是，则沿关键 DevOps 步骤拆分故事。
8. 如果上述标准都不适用，请考虑使用微小的发现行动（TAD）来解开未知因素、假设和复杂性，并相应地完善故事。

## 用户故事格式提示词模板：

_以下格式结合了 Mike Cohn 用户故事格式，由 Gherkin 风格的验收标准增强。_

### 用户故事 [用户故事编号 ID]：

- **摘要**：[简短、易记、人类可读的故事标题，说明我们如何为用户角色提供价值]

#### 用例：
- **作为** [用户名称（如果可用），否则为用户角色，否则为用户角色或职位]，
- **我想要** [用户采取行动以达到结果]，
- **以便** [用户期望的结果]。

#### 验收标准：

(_注意 1：Given 是前提条件，所以不要限制为只有 3 个 Given。_
 _注意 2：应该只有一个"When"和一个"Then"声明，与用例的"我想要"行动和"以便"结果保持一致。_
 _注意 3：多个"when"和"then"是故事必须拆分的良好指标。_)

- **场景**：[简短、人类可读的用户场景，说明我们如何为用户角色提供价值]
- **Given**：[初始背景或前提条件]
- **and Given**：[基于用户背景的额外背景或前提条件]
- **and Given** [根据需要支持"用例"的额外背景或前提条件]
- **and Given** [确保"When"事件可以发生的额外用户界面焦点背景或前提条件]
- **and Given** [确保"Then"结果被交付的额外结果焦点背景或前提条件]
- **When**：[发生与用例行动相连的事件]
- **Then**：[与用例结果相连的预期结果]
